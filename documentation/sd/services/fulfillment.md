# Fulfillment service

After payment is captured, fulfillment turns "the user paid for these
apps" into "the user can install these apps". For every app in an
order, fulfillment:

1. Mints a license via `license_hook.issue_license`. The license is the
   user's permanent proof of purchase.
2. Issues a per-platform download URL via `ios_distribution_hook` and
   `macos_distribution_hook`.
3. Records a receipt under the `order_id` and adds the app to the
   user's `library`.

A user's library is the union of every app they have a license for —
this is what "Restore purchases" reads from on a new device.

## Endpoints

| Method | Path                              | Notes |
| ------ | --------------------------------- | ----- |
| POST   | `/fulfill`                        | Body: `{order_id, user_id, app_ids[]}`. |
| GET    | `/receipts/<order_id>`            | One receipt with all entitlements. |
| GET    | `/users/<user_id>/library`        | Map of `app_id → license`. |

## Hooks

* `ios_distribution_hook.issue_download(app_id, user_id)` →
  `{platform, url, expires_at}`. URL is a stub.
* `macos_distribution_hook.issue_download(app_id, user_id)` → same shape.
* `license_hook.issue_license(app_id, user_id, order_id)` →
  `{license_id, app_id, user_id, order_id, kind: "perpetual"}`.

The hooks are intentionally thin so a real platform can be plugged in
behind them without changing the rest of the service.

## Why this is reused by PearCare

A "replacement" claim in PearCare is, end-to-end, equivalent to an
unpaid re-purchase: a fresh license keyed to the same `app_id`. The
PearCare claim service calls `POST /fulfill` directly, passing
`order_id = "replacement-<claim_id>"` so the receipt is distinguishable
from a real purchase. The user's library accumulates the new license
just as if they had bought the app twice.

## Known limitations

* Download URLs are deterministic strings; nothing actually serves them.
* No revocation. Cancelling or refunding does not invalidate the license.
* `expires_at` on download URLs is an hour from issue but is not
  enforced anywhere.

## Data model

A licensing agreement within the fulfillment process has the following primary characteristics and essential properties:

* The Fulfillment Service (FUL) plays a key role in managing entitlements and replacements [S2](../database/cart-db.md), [S3](../database/pearcare-claim-db.md).
* FUL processes replacement orders and sends entitlements in response to POST requests from the CL service [S3](../database/pearcare-claim-db.md), [S4](../architecture/data-flow-purchase.md).
* When a claim resolves to a replacement, FUL reuses its logic from PearCare's claim service [S2](../database/cart-db.md).
* The licensing agreement is part of the fulfillment process, which involves minting licenses and per-platform download URLs [S1](../database/cart-db.md).

The essential properties of a licensing agreement within the fulfillment process include:

* A clear separation of concerns between services, enabling each service to be developed and maintained independently [S6](../pearcare/overview.md).
* A modular architecture that facilitates scalability and flexibility in managing PearCare claims [S1](../database/cart-db.md).
* The use of POST requests for replacement entitlements, which triggers FUL to process replacement orders and send entitlements in response [S3](../database/pearcare-claim-db.md), [S4](../architecture/data-flow-purchase.md).

Note: Further detail regarding the specific data stored in the account database for each user is TBD [S7](../database/pearcare-claim-db.md).

## Observability

The fulfillment service's behavior can be observed through various monitoring and logging mechanisms. The catalog service generates event logs related to interactions with the review service via the catalog API [S2](catalog.md). Additionally, request traces related to cart management are generated by the catalog service [S5](catalog.md).

The account database stores user-specific data, including cart information, which is accessed by various services, including the order service and the fulfillment service [S4](../database/pearcare-claim-db.md), [S6](../database/account-db.md), [S7](../database/review-db.md). The account database follows a state machine for "pending → paid → fulfilled" transactions, which implies that the cart database likely uses a consistent state model to manage its data [S3](../database/pearcare-claim-db.md).

The Fulfillment Service (FUL) processes replacement orders and sends entitlements in response to POST requests from the CL service [S3](../database/pearcare-claim-db.md), [S7](../database/review-db.md). However, further detail regarding the specific data stored in the account database for each user is TBD [further detail TBD].

## Open Questions

Unresolved issues or design decisions that need further discussion or clarification in the fulfillment service include:

* The exact duration of data retention for orders in the `failed` state is not explicitly stated [S6](../runbooks/pearcare-fraud.md).
* Automatic refunds are not handled upon fulfillment failure, requiring human intervention to run a POST request to the Order Service (ORD) with the order ID for refund [S10].
* The consistency model used by the cart db and the account database, as well as the specific data stored in these databases for each user, is unclear [S2](../runbooks/pearcare-fraud.md), [S5](../runbooks/incident-response.md).

Further detail is needed regarding the implementation of automatic refunds upon fulfillment failure.
