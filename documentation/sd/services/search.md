# Search service

Lightweight full-text search over the catalog. Indexes name, developer,
and description.

## Endpoint

`GET /search?q=...&category=...&price=free|paid&limit=N`

| Param      | Required | Notes |
| ---------- | -------- | ----- |
| `q`        | no       | Free-text query. Empty `q` returns the most popular apps. |
| `category` | no       | Exact category match (case-insensitive). |
| `price`    | no       | `free` for `price_cents == 0`, `paid` for the rest. |
| `limit`    | no       | Defaults to 20. |

## Ranking

Tokens come from a permissive `[a-z0-9]+` regex over lowercased text.
For each query token we look at every haystack token from `name +
developer + description`:

* exact match: +3
* prefix match: +1

Apps are sorted by `(score desc, rating_count desc)`. The popularity
tiebreaker matters when many apps match a generic query like "weather"
or "todo".

## Index freshness

The index is rebuilt by calling catalog-svc on every search. This is
fine for fifty apps; in a real catalog we'd subscribe to a change feed.
The contract here is "search reflects whatever catalog returned right
now".

## Not supported

* Misspellings. `wether` will not find "Weather".
* Synonyms. `runner` does not match "jogging".
* Personalization. The same query returns the same results for everyone.

## Downstream services

The Search service calls the **Order** and **Payment** services to fulfill its responsibility. 

When a user initiates a checkout, the Order service retrieves the cart contents from the Cart Service [S4](order.md). The Order service then interacts with the Payment Service to authorize the payment for the order [S3](order.md) and retrieve relevant data [S5](../database/payment-db.md).

The Search service likely relies on the Order service's state machine for "pending → paid → fulfilled" transactions to determine the status of a user's orders. However, further detail regarding how the Search service uses this information is TBD.

Note that the Payment Service wraps three provider hooks (stripe, paypal, pearcard) [S6](../architecture/overview.md), and the Fulfillment Service mints licenses and per-platform download URLs [S7](../database/cart-db.md). The Review service owns ratings and free-text reviews and pushes aggregate ratings back into catalog so search can use them [S6](../architecture/overview.md).

The Search service may also rely on the Catalog service to retrieve prices or other relevant data for a user's orders [S2](../database/payment-db.md).

## Observability

The Search service emits or records the following error logs, key performance metrics, and request/response traces to monitor its execution time, throughput, and latency:

* Error logs related to interactions with the catalog service via the catalog API [S1](fulfillment.md)
* System metrics from the catalog service, which can provide insights into the performance of the catalog service, such as CPU usage or memory consumption [S3](catalog.md) (further detail TBD)
* Request traces from the catalog service, which can help identify bottlenecks in cart management processes [S5](catalog.md)

Note that the specific data stored in the cart database for each user is unclear (further detail TBD) [S4](catalog.md), but the Cart service stores information about items a user has added to their cart, including `user_id`, `items`, and other relevant data [S9](../database/catalog-db.md).

The catalog service generates event logs related to interactions with the review service via the catalog API [S3](../database/payment-db.md) and system metrics that are likely produced by the catalog service due to its interactions with other services [S4](catalog.md). The catalog service also owns the catalog database and is responsible for managing user carts, which implies that request traces related to cart management are generated by the catalog service [S5](catalog.md).

The Search service can use event logs from the catalog service to monitor interactions between the review service and the catalog service. System metrics from the catalog service can provide insights into the performance of the catalog service. Request traces from the catalog service can help identify bottlenecks in cart management processes.

Typical query shapes for accessing apps in the catalog database include retrieving app information by ID, such as `SELECT * FROM catalog WHERE app_id = ?` [S3](catalog-db.md). The most frequently accessed fields are likely to be `app_id`, `price_cents`, and other attributes related to product information [S2](../architecture/overview.md).

The catalog service interacts with other services, including the Review service, which writes data to the catalog database via the catalog API [S3](../database/catalog-db.md). The catalog database schema likely includes attributes such as `app_id`, `price_cents`, and `items` [S5](../database/catalog-db.md).

The Search service can use this information to monitor its execution time, throughput, and latency by analyzing error logs, system metrics, and request/response traces from the catalog service.

There are several open problems or areas requiring subject matter expert input to enhance the Search functionality. 

The Catalog service has additional open questions or areas that need SME input [S1](catalog.md). The catalog database also has remaining open questions or areas that need SME input [S2](../database/catalog-db.md). Furthermore, refining the triage rules to better categorize and prioritize customer issues based on keywords could benefit from SME input [S3](../runbooks/pearcare-fraud.md).

Additionally, there are areas where further detail is TBD regarding indexing strategies for the databases [S4](../database/review-db.md), backup strategies for the databases [S5](../database/review-db.md), and index and backup strategies [S5](../database/review-db.md). The payment database has open questions related to it that need SME input [S6](../database/payment-db.md). 

The review service has several open questions and unresolved issues that require SME input, including handling automatic refunds upon fulfillment failure [S2](../database/catalog-db.md) and clarifying what non-goals or out-of-scope work are explicitly excluded from this purchase data flow [S1](catalog.md).

Improving payment processing by handling declined payments more efficiently and enhancing order fulfillment by automating refunds upon fulfillment failure are areas that require SME input. The system's architecture has been driven by modularity and decoupling of services, allowing for independent development and maintenance [S4](../database/review-db.md). However, the exact duration of data retention for orders in the `failed` state is not explicitly stated [S6](../database/payment-db.md), which may be an area where SME input is required.

These areas require SME input to clarify the consistency model, the specific data stored in the cart db, and the data stored in the account database [S11](../database/catalog-db.md).
