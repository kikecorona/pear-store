## What a good telemetry doc looks like

See `order.md`. It covers, in order:

1. **What the service is and what it instruments.**
2. **Span inventory** — name, when it fires, attributes.
3. **Metric inventory** — name, kind, attributes, what each value means.
4. **End-to-end trace example** — one full happy-path checkout and one
   payment-declined trace, both stitched via `traceparent`.
5. **Failure semantics** — which conditions land as `status=ERROR`
   vs `status=OK with attributes.status=failure`, and why.
6. **Known gaps and follow-ups.**

Anything missing one of those sections is a candidate for the gap
agent to fill in.

## Details

A telemetry system consists of several key components that work together to provide real-time monitoring and insights. These components include:

* **Data Collection**: Event logs, system metrics, and request traces are generated by various services, including the catalog service (which emits event logs related to interactions with the review service via the catalog API [S1](../services/review.md)), the account service (which generates audit records, performance indicators, and distributed request data for monitoring and troubleshooting purposes [S6](../services/account.md)), and the cart service (which owns the cart database and is responsible for managing user carts [S8](../database/payment-db.md)). System metrics from the catalog service can provide insights into the performance of the catalog service, such as CPU usage or memory consumption [further detail TBD] [S3](../services/order.md).
* **Data Transmission**: The telemetry system transmits collected data to a central location for analysis. This may involve interactions between services, such as the account service and the catalog service, which exchange data to manage user carts and retrieve product information [S6](../services/account.md).
* **Data Analysis**: The telemetry system analyzes transmitted data to provide real-time monitoring and insights. This may involve analyzing event logs, system metrics, and request traces to identify bottlenecks in cart management processes [S1](../services/review.md) or to monitor interactions between services.

The primary functions of a telemetry system include:

* Monitoring interactions between services
* Providing insights into service performance
* Identifying bottlenecks in business processes

The components of a telemetry system interoperate as follows:

* The catalog service generates event logs and system metrics, which are transmitted to the account service for analysis.
* The account service generates audit records, performance indicators, and distributed request data, which are analyzed to provide real-time monitoring and insights.
* The cart service manages user carts and exchanges data with the account service to retrieve product information.

Overall, a telemetry system provides real-time monitoring and insights by collecting, transmitting, and analyzing data from various services.
